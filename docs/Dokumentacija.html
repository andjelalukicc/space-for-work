1. Uvod

Tema ovog projekta je izgradnja sistema za rezervaciju prostorija u coworking prostoru, realizovana kroz mikroservisnu arhitekturu. Cilj projekta bio je da se u praksi primene koncepti integracije sistema koji su obrađivani na predavanjima, uključujući različite tipove komunikacije između servisa, automatizaciju procesa razvoja i monitoring aplikacije.

Projekat je razvijen korišćenjem Node.js frejmvorka NestJS i pokreće se lokalno putem Docker Compose-a. Sistem se sastoji od pet nezavisnih mikroservisa koji međusobno komuniciraju putem REST API-ja, message broker-a RabbitMQ i reaktivnih stream-ova (SSE). Svaki servis ima jasno definisanu odgovornost i sopstvene podatke.

2. Arhitektura implementiranog sistema

2.1 Pregled sistema

Sistem je implementiran kao skup od pet nezavisnih mikroservisa. Klijent nikada ne komunicira direktno sa pojedinim servisima — sve zahteve prima API Gateway koji ih prosleđuje odgovarajućem servisu. Ovakav pristup omogućava da svaki servis može da se razvija, testira i skalira nezavisno od ostalih.

2.2 Uloge mikroservisa

API Gateway (port 3000) — jedina tačka ulaska u sistem. Prima sve zahteve od klijenta, proverava JWT token na zaštićenim rutama i prosleđuje zahtev odgovarajućem servisu. Gateway dodaje x-user-id header u interne zahteve, kako bi servisi znali koji korisnik je ulogovan bez ponovnog dekodiranja tokena.

User Service (port 3001) — zadužen je za upravljanje korisnicima. Podržava registraciju, login i pregled profila. Lozinka se čuva u hashovanom obliku (bcrypt), a nakon uspešnog login-a klijentu se vraća JWT token koji važi 24 sata.

Room Service (port 3002) — upravlja podacima o prostorijama. Pri prvom pokretanju automatski upisuje 8 prostorija u bazu: 2 meeting room-a i 6 phone booth-ova. Pored REST endpointa, implementira SSE stream koji klijentima šalje obaveštenja o promeni dostupnosti u realnom vremenu.

Booking Service (port 3003) — najsloženiji deo sistema, sadrži svu poslovnu logiku vezanu za rezervacije. Pre kreiranja rezervacije sistem proverava: da li je prostorija slobodna u željenom terminu, da li korisnik već ima rezervaciju koja se preklapa i da li je dostignut dnevni limit od tri rezervacije. Tek ako su svi uslovi ispunjeni, rezervacija se kreira i šalje se event putem RabbitMQ-a.

Notification Service (port 3004) — radi kao hibridna aplikacija, istovremeno prima HTTP zahteve i sluša poruke sa RabbitMQ reda. Kada Booking Service pošalje event booking_created ili booking_cancelled, ovaj servis kreira obaveštenje i čuva ga u bazi. Korisnik može da pročita svoja obaveštenja putem REST endpointa.

3. Tipovi komunikacije između servisa

3.1 Sinhrona komunikacija — REST API

REST API koristi se za komunikaciju između API Gateway-a i svih internih servisa, kao i između Booking Service-a i Room Service-a pri proveri dostupnosti prostorije. Radi se o sinhronoj komunikaciji — servis šalje zahtev i čeka odgovor pre nego što nastavi sa obradom. Ovaj tip komunikacije pogodan je za situacije u kojima je odgovor neophodan za dalju obradu zahteva, kao što je slučaj kod provere slobodnih termina pre kreiranja rezervacije.

3.2 Asinhrona komunikacija — RabbitMQ

RabbitMQ message broker koristi se za komunikaciju između Booking Service-a i Notification Service-a. Nakon kreiranja ili otkazivanja rezervacije, Booking Service objavljuje event na RabbitMQ red, a Notification Service ga nezavisno preuzima i obrađuje.

Ova komunikacija je asinhrona — Booking Service ne čeka odgovor od Notification Service-a i ne mora da zna da on uopšte postoji. Ovakav pristup smanjuje međuzavisnost servisa i povećava otpornost sistema na greške: čak i ako Notification Service nije dostupan, rezervacija će biti uspešno kreirana, a obaveštenje će biti isporučeno čim servis ponovo postane aktivan.

// Booking Service — slanje eventa nakon uspešne rezervacije
this.rabbitClient.emit('booking_created', {
  userId:    booking.userId,
  roomId:    booking.roomId,
  date:      booking.date,
  startTime: booking.startTime,
  endTime:   booking.endTime,
});

// Notification Service — prijem eventa
@EventPattern('booking_created')
async handleBookingCreated(data: BookingCreatedDto) {
  await this.notificationsService.create(data);
}
Isečak koda 1 — Asinhrona komunikacija putem RabbitMQ

3.3 Reaktivna komunikacija — SSE (Server-Sent Events)

Server-Sent Events implementiran je u Room Service-u i omogućava klijentu da prima obaveštenja o promeni dostupnosti prostorija u realnom vremenu, bez potrebe da periodično šalje zahteve serveru. Klijent otvori jednu HTTP konekciju ka endpointu /rooms/availability/stream i server mu šalje podatke čim se stanje promeni. Ovaj pristup je efikasniji od klasičnog pollinga i čini sistem reaktivnim u smislu distribucije podataka u realnom vremenu.

4. Baza podataka

Koristi se jedna instanca PostgreSQL baze podataka, ali svaki servis radi isključivo sa sopstvenim tabelama. User Service koristi tabelu users, Room Service tabelu rooms, Booking Service tabelu bookings, a Notification Service tabelu notifications. Na ovaj način je postignut logički princip izolacije podataka karakterističan za mikroservise, uz jednostavnost jedne infrastrukturne instance baze.

Mapiranje između TypeScript klasa i tabela u bazi odrađuje TypeORM biblioteka, koja automatski kreira tabele na osnovu definisanih entity klasa. Ovaj pristup se naziva code-first — schema baze se definiše u kodu, a ne ručno u SQL-u.

6. Kontejnerizacija — Docker

6.1 Dockerfile

Svaki mikroservis ima sopstveni Dockerfile koji koristi multi-stage build pristup. Ovaj pristup podrazumeva dve faze izgradnje slike: u prvoj fazi (builder) kompajlira se TypeScript kod u JavaScript, a u drugoj fazi (production) u finalnu sliku kopira se samo kompajlirani kod i produkcione zavisnosti. Razvojni alati i izvorni kod ne ulaze u finalnu sliku, što je čini manjom i bezbednijom.

# Faza 1: kompajliranje TypeScript koda
FROM node:22-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# Faza 2: produkciona slika (bez razvojnih alata)
FROM node:22-alpine AS production
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
CMD ["node", "dist/main"]
Isečak koda 2 — Multi-stage Dockerfile (primer za jedan servis)

6.2 Docker Compose

Svi servisi definisani su i orkestrirani u jednom docker-compose.yml fajlu. Ukupno se pokreće devet kontejnera: pet NestJS mikroservisa, PostgreSQL, RabbitMQ, Prometheus i Grafana. Svi kontejneri nalaze se u zajedničkoj Docker mreži coworking-network, što im omogućava međusobnu komunikaciju putem naziva servisa umesto IP adresa.

Za PostgreSQL i RabbitMQ definisani su healthcheck-ovi, a NestJS servisi su konfigurisani da sačekaju da infrastrukturni servisi budu potpuno pokrenuti pre nego što sami krenu sa radom. Na taj način se sprečavaju greške koje bi nastale pri pokušaju konekcije na bazu koja još nije inicijalizovana.

services:
  postgres:
    image: postgres:16-alpine
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]

  booking-service:
    build: ./booking-service
    depends_on:
      postgres:
        condition: service_healthy
      rabbitmq:
        condition: service_healthy
    ports:
      - "3003:3003"
Isečak koda 3 — Isečak iz docker-compose.yml sa healthcheck zavisnošću

Aplikacija se pokreće jednom komandom: docker compose up -d. Nakon pokretanja, svih devet kontejnera je u statusu healthy.

7. CI pipeline — GitHub Actions

Svaki push na GitHub i svaki otvoreni Pull Request automatski pokreće CI (Continuous Integration) pipeline definisan u fajlu .github/workflows/ci.yml. Cilj CI pipeline-a je da se automatski proveri ispravnost koda pre nego što bude spojen u glavnu granu.

Pipeline koristi matrix strategiju, što znači da se isti skup koraka izvršava paralelno za svih pet servisa. Na taj način celokupni proces traje znatno kraće nego kada bi se servisi obrađivali jedan po jedan.

strategy:
  matrix:
    service:
      - api-gateway
      - user-service
      - room-service
      - booking-service
      - notification-service

steps:
  - name: Checkout koda
    uses: actions/checkout@v4

  - name: Setup Node.js 22
    uses: actions/setup-node@v4

  - name: Instalacija zavisnosti
    run: npm install

  - name: ESLint — statička analiza koda
    run: npx eslint src/

  - name: Build — kompilacija TypeScript-a
    run: npm run build

  - name: Pokretanje unit testova
    run: npx jest --passWithNoTests

  - name: Build Docker image
    run: docker build -t coworking-${{ matrix.service }} .
Isečak koda 4 — CI pipeline konfiguracija (GitHub Actions)

Ako bilo koji korak u pipeline-u ne prođe — bilo da se radi o ESLint grešci, neuspelom testu ili grešci pri build-u — ceo pipeline pada i Pull Request ne može biti spojen u main granu. Na taj način garantuje se da samo ispravan, testiran kod ulazi u produkciju.

8. Statička analiza koda — ESLint

ESLint je konfigurisan za svih pet servisa korišćenjem flat config formata (eslint.config.mjs), koji je standard od ESLint v9. Integrisan je TypeScript ESLint plugin zajedno sa Prettier alatom za formatiranje koda.

Pravila su prilagođena specifičnostima projekta. U API Gateway-u su isključena no-unsafe-* pravila jer proxy pattern inherentno zahteva rad sa any tipovima pri prosleđivanju zahteva. Za sve ostale servise primenjena su standardna TypeScript ESLint pravila. Rezultat statičke analize: nula grešaka u svih pet servisa. ESLint se izvršava kao prvi korak u CI pipeline-u, pre build-a i testova.

9. CD pipeline i deployment

CD (Continuous Delivery) pipeline definisan je u fajlu .github/workflows/cd.yml i pokreće se automatski kada se kod spoji u main granu, odnosno nakon merge-a Pull Request-a.

Pipeline se izvršava u tri uzastopna koraka. U prvom koraku ponovo se pokreću svi testovi za svih pet servisa — ovo je pre-deploy verifikacija koja osigurava da nijedan neispravni kod ne dođe u produkciju. U drugom koraku prave se Docker slike sa dva taga: latest koji uvek pokazuje na poslednju verziju i SHA hash trenutnog commit-a koji omogućava tačnu identifikaciju svake verzije. U trećem koraku vrši se verifikacija kompletnog Docker Compose stack-a pokretanjem svih devet kontejnera i proverom health endpointa API Gateway-a.

Aplikacija se deployi lokalno putem Docker Compose-a. API Gateway je dostupan na adresi http://localhost:3000, a Swagger dokumentacija na http://localhost:3000/api.

10. Monitoring — Prometheus i Grafana

10.1 Prometheus

Svaki mikroservis eksponuje /metrics endpoint koji Prometheus periodično čita svakih 15 sekundi. Metrike se prikupljaju korišćenjem prom-client biblioteke i uključuju broj HTTP zahteva, vreme obrade (latency) po endpointu, broj grešaka i broj aktivnih konekcija.

# Prometheus konfiguracija — scraping svakih 15 sekundi
scrape_configs:
  - job_name: 'api-gateway'
    scrape_interval: 15s
    static_configs:
      - targets: ['api-gateway:3000']

  - job_name: 'user-service'
    static_configs:
      - targets: ['user-service:3001']

  - job_name: 'booking-service'
    static_configs:
      - targets: ['booking-service:3003']
Isečak koda 5 — Prometheus konfiguracija za scraping mikroservisa

10.2 Grafana

Grafana je konfigurisana sa automatskim provisioning-om — datasource i dashboard učitavaju se automatski pri pokretanju, bez ručne konfiguracije. Dashboard sadrži pet panela koji prikazuju:

  • HTTP request rate po servisu — prikazuje aktivnost svakog servisa u realnom vremenu
  • Latency percentili p50, p95 i p99 — koliko dugo traje obrada zahteva
  • Error rate za 4xx i 5xx odgovore — praćenje grešaka u sistemu
  • Aktivne konekcije — broj aktivnih veza sa svakim servisom
  • Gauge prikaz trenutnog request rate-a — brzi pregled opterećenja sistema

Grafana je dostupna na http://localhost:3100 sa podrazumevanim kredencijalima (admin/admin), a Prometheus na http://localhost:9090.

11. Zaključak

U okviru ovog projekta implementiran je kompletan sistem za rezervaciju prostorija u coworking prostoru kao mikroservisna aplikacija. Sistem obuhvata pet NestJS mikroservisa koji međusobno komuniciraju putem tri različita tipa komunikacije: sinhroni REST API, asinhroni RabbitMQ i reaktivni SSE stream.

Projekat je u potpunosti pokriven automatskim testovima — ukupno 74 testa (40 unit i 34 E2E), svi prolaze. Celokupna infrastruktura je kontejnerizovana putem Docker Compose-a koji pokreće devet kontejnera. Automatizovani CI/CD pipeline putem GitHub Actions obezbeđuje da svaki Push i Pull Request prođe kroz statičku analizu koda, kompilaciju, testove i verifikaciju Docker build-a pre nego što bude spojen u glavnu granu.

Monitoring sistema implementiran je putem Prometheus i Grafana alata koji u realnom vremenu prikazuju metrike svakog servisa. Sve faze projekta su završene i aplikacija se uspešno pokreće lokalno.
